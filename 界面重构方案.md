# CodexMultiLauncher 界面与架构重构方案

## 1. 背景

当前 `CodexMultiLauncher` 的主要功能集中在 `src/codex_multi_launcher/app.py`：

- 界面构建、账号列表渲染、弹窗和进度条都在同一个窗口类中。
- 配置读写、账号目录迁移、程序副本更新、会话同步、记忆同步、诊断报告也在同一个文件中。
- 项目已切换到 Electron + React 界面，后续重点是收敛桌面壳、Python 后端和打包链路。

目标不是简单换皮，而是把系统拆成“核心业务”和“现代界面”两层，让后续功能扩展、打包和排错更稳定。

## 2. 参考项目结构

参考 `D:\rwkj\wechat_rpa_helper_pc_client` 的整体分层：

- 根目录负责桌面应用工程、打包配置和前端依赖。
- `src/main` 负责 Electron 主进程、窗口、系统能力、进程管理。
- `src/preload` 负责安全桥接，只暴露允许页面调用的接口。
- `src/renderer` 负责页面、组件、状态和接口封装。
- `python` 负责 Python 后端能力和打包产物。
- `resources` 负责图标、可执行后端、静态资源等运行时资源。

`CodexMultiLauncher` 不需要完整照搬该项目的大量业务模块和复杂依赖，应采用同样的分层思想，但实现轻量版本。

## 3. 重构目标

### 3.1 用户体验目标

- 首页从“按钮堆叠”调整为“账号控制台”。
- 账号运行状态、异常状态、程序副本状态、配置健康状态直接展示。
- 新增账号、启动账号、改名、删除、打开目录等操作层级清晰。
- `CodexProfiles` 迁移、程序副本更新、同步准备等耗时任务显示可读进度。
- 诊断报告从纯文本弹窗升级为结构化状态页。
- 设置项从主页面抽离，降低主页面信息密度。

### 3.2 工程目标

- 保留现有 Python 核心逻辑，避免一次性重写导致账号隔离和迁移逻辑回退。
- 先拆业务模块，再接入 Electron 界面。
- 收敛到 Electron 入口，Python 侧只保留桥接后端和核心服务。
- 新界面通过统一命令接口调用 Python 能力，不直接操作配置和系统路径。
- 打包流程明确区分前端桌面壳、Python 后端和最终安装包。

## 5. 模块职责设计

### 5.1 Python 核心模块

`config_store.py`

- 读取配置。
- 写入配置。
- 维护最近两份可用备份。
- 主配置损坏时从备份恢复。

`codex_source.py`

- 自动识别微软商店版 Codex。
- 读取 Codex 版本信息。
- 判断账号程序副本是否需要更新。

`profile_service.py`

- 新增账号。
- 删除账号。
- 账号改名。
- 导入原本账号。
- 启动指定账号。
- 计算账号目录、`CodexHome`、用户数据目录。

`migration_service.py`

- 迁移整个 `CodexProfiles` 根目录。
- 清理可丢弃临时目录。
- 跳过 junction 等特殊目录。
- 按文件块汇报复制进度。

`sync_service.py`

- 准备会话同步。
- 准备记忆同步。
- 合并 SQLite 数据库。
- 隔离损坏数据库。

`diagnostics.py`

- 生成基础诊断信息。
- 检查账号目录、程序副本、`CodexHome`、项目注册数、记忆数。
- 输出结构化诊断结果，供新界面渲染。

### 5.2 Python 桥接模块

`bridge/commands.py` 对外提供统一命令：

```text
get_app_state
set_codex_path
refresh_codex_source
list_profiles
create_profile
rename_profile
delete_profile
launch_profile
set_profile_root
migrate_profile_root
set_session_sync
set_memory_sync
get_diagnostics
open_path
```

所有命令返回统一结构：

```json
{
  "ok": true,
  "data": {},
  "error": ""
}
```

耗时任务额外返回任务标识，由界面轮询或订阅进度：

```json
{
  "taskId": "profile-root-migration",
  "phase": "copying",
  "percent": 42.5,
  "detail": "正在复制 state_5.sqlite"
}
```

### 5.3 Electron 主进程

`src/main/index.ts`

- 创建主窗口。
- 管理开发环境和生产环境资源路径。
- 启动或连接 Python 后端。

`src/main/ipc.ts`

- 注册页面可调用的接口。
- 统一处理成功和失败。
- 禁止页面直接执行任意命令。

`src/main/python/launcherBackend.ts`

- 启动 `resources/main.exe` 或开发环境下的 Python 脚本。
- 管理后端进程生命周期。
- 转发命令请求和进度事件。

`src/main/windows`

- 管理主窗口。
- 后续如需要，可扩展诊断窗口、设置窗口。

### 5.4 预加载层

`src/preload/index.ts`

- 暴露 `window.launcherApi`。
- 只提供白名单接口。
- 不暴露 Node.js 原生能力给页面。

建议接口：

```ts
window.launcherApi = {
  invoke(command, payload),
  onTaskProgress(callback),
  offTaskProgress(callback)
}
```

### 5.5 渲染层页面

`Dashboard`

- 总览页面。
- 展示账号数量、运行中账号、异常账号、Codex 版本、账号根目录。
- 提供常用账号快速启动。

`Profiles`

- 账号管理页。
- 账号列表使用卡片或表格。
- 每个账号展示状态、目录、程序副本、项目数、记忆数。
- 操作包括启动、改名、删除、打开目录。

`Settings`

- Codex 程序路径。
- 账号根目录。
- 会话同步开关。
- 记忆同步开关。
- 同步目录展示和打开。

`Diagnostics`

- 结构化诊断页。
- 按基础信息、账号状态、同步状态、异常建议分组。
- 支持复制诊断报告。

`TaskProgress`

- 全局任务进度组件。
- 展示迁移、复制、同步准备等任务。
- 长任务可显示当前文件、百分比、已处理大小。

## 6. 界面设计方向

### 6.1 主窗口布局

```text
┌──────────────────────────────────────────────┐
│ Codex 多账号切换器        设置  诊断  刷新     │
├───────────────┬──────────────────────────────┤
│ 账号列表       │ 账号详情                     │
│               │                              │
│ ● 工作账号     │ 状态：运行中                  │
│ ○ 测试账号     │ 程序副本：已同步              │
│ △ 临时账号     │ CodexHome：正常              │
│               │ 项目数：12  记忆数：240       │
│ + 新增账号     │                              │
├───────────────┴──────────────────────────────┤
│ 任务状态：正在迁移 CodexProfiles... 42%        │
└──────────────────────────────────────────────┘
```

### 6.2 视觉原则

- 主页面只放最高频操作：选择账号、启动账号、查看状态。
- 设置类操作统一放到设置页。
- 诊断类操作统一放到诊断页。
- 按钮数量减少，次要操作放入更多菜单。
- 状态使用文字加颜色标识，例如运行中、就绪、需处理、异常。
- 弹窗只用于确认或阻断性错误，不再承载大量常规信息。
- 大量路径文本默认省略显示，悬停或点击后查看完整路径。

## 7. 分阶段实施计划

### 第一阶段：拆分 Python 核心

目标：

- 不改变用户可见行为。
- 从 `app.py` 中拆出核心模块。
- Electron 界面通过桥接命令调用拆分后的服务。

改动范围：

- 新增 `src/codex_multi_launcher/core`。
- 新增 `src/codex_multi_launcher/bridge`。
- 逐步迁移配置、账号、迁移、同步、诊断逻辑。

验证：

```bash
uv run python -m py_compile ./src/codex_multi_launcher/app.py
uv run python -m py_compile ./src/codex_multi_launcher/core/config_store.py
uv run python -m py_compile ./src/codex_multi_launcher/core/profile_service.py
```

### 第二阶段：建立 Electron 工程骨架

目标：

- 增加 `package.json`、`electron.vite.config.ts`、`src/main`、`src/preload`、`src/renderer`。
- 先做空壳窗口和基础路由。
- 不接入全部业务，只验证窗口、构建和开发启动流程。

建议依赖：

- Electron：桌面应用框架。
- React：界面库。
- Vite：前端构建工具。
- TypeScript：类型化脚本语言。
- Ant Design：界面组件库。
- lucide-react：图标库。

验证：

```bash
yarn install
yarn dev
yarn build
```

### 第三阶段：接入 Python 后端命令

目标：

- Electron 主进程可以调用 Python 命令。
- 页面可以读取账号列表、基础状态和诊断信息。
- 支持启动账号。

优先接入命令：

```text
get_app_state
list_profiles
launch_profile
get_diagnostics
open_path
```

验证：

- 新界面能展示真实账号。
- 新界面能启动指定账号。
- 新界面的诊断结果与核心服务输出一致。

### 第四阶段：迁移完整页面

目标：

- 完成 Dashboard、Profiles、Settings、Diagnostics。
- 支持新增、改名、删除、导入原账号。
- 支持 Codex 路径识别和刷新。
- 支持同步开关。
- 支持账号根目录迁移和进度展示。

验证：

- 老功能逐项对照通过。
- 迁移目录时进度可见。
- 账号运行中时更新程序副本的阻断逻辑仍然有效。
- 配置损坏恢复逻辑仍然有效。

### 第五阶段：打包和旧入口收敛

目标：

- Python 后端打包为 `resources/main.exe`。
- Electron 打包为最终桌面应用。
- Python 后端打包前先将 `bridge`、`core` 模块编译为 `pyd`。

验证：

```bash
yarn build
yarn build:win
```

## 8. 风险与处理策略

### 8.1 账号隔离风险

风险：

- 新界面调用链变化后，可能影响 `APPDATA`、`LOCALAPPDATA`、`CODEX_HOME`、`--user-data-dir` 的隔离。

策略：

- 启动逻辑先原样迁入 `profile_service.py`。
- 新界面只调用 `launch_profile`，不拼接环境变量。
- 诊断页继续展示每个账号的关键目录。

### 8.2 目录迁移风险

风险：

- `CodexProfiles` 跨盘迁移涉及大量文件、临时目录、junction 和运行中进程。

策略：

- 复用现有迁移逻辑。
- 进度事件只做展示，不改变复制策略。
- 迁移前继续清理 `CodexHome\.tmp`。
- 继续跳过 reparse point。

### 8.3 同步数据库风险

风险：

- 会话和记忆同步涉及 SQLite，直接共享或复制不当可能损坏数据库。

策略：

- 复用现有 SQLite 健康检查、备份、合并和隔离逻辑。
- 新界面只展示同步状态，不直接操作数据库文件。

### 8.4 打包复杂度风险

风险：

- Electron 加 Python 后端会增加打包链路。

策略：

- 第一阶段不引入新打包。
- 第二阶段只验证 Electron 空壳。
- 后端可执行文件稳定后再合并最终打包。

## 9. 推荐执行顺序

1. 评审本方案，确认是否采用 Electron 加 Python 的轻量结构。
2. 拆分 Python 核心逻辑，保持桥接命令可用。
3. 建立 Electron 工程骨架，先跑通空壳。
4. 接入账号列表、启动账号、诊断报告。
5. 完成设置页和迁移进度。
6. 对照既有功能逐项验收。
7. 调整打包流程，生成新版可执行文件。

- 不在拆分阶段改交互和视觉。

## 11. 结论

推荐采用“保留 Python 核心，新增 Electron 加 React 界面”的重构路线。

这条路线既能解决当前页面难看的问题，也能避免把已经验证过的账号隔离、迁移、同步逻辑全部推倒重写。参考 `wechat_rpa_helper_pc_client` 的目录边界即可，不需要复制它的全部业务复杂度。
