# 参考项目分析

本文记录对 `codex-tools-2.0.5` 的只读分析，以及其中适合借鉴到 CodexMultiLauncher 的工程做法。

## 参考项目概况

`codex-tools-2.0.5` 是一个面向 Codex 多账号管理的桌面工具，技术栈为：

- React：前端界面。
- Tauri：桌面应用壳。
- Rust：本地后端和系统能力。

它的功能范围比 CodexMultiLauncher 更大，包含：

- 多账号导入、重授权、删除、别名管理。
- Codex 登录态切换。
- 用量刷新和成本统计。
- 本地 OpenAI 兼容 API 反代。
- Cloudflare 隧道公网访问。
- 远程 Linux 反代部署。
- 状态栏、自动更新、多语言界面。

CodexMultiLauncher 的定位更轻量：只负责在 Windows 上保存多份 Codex 账号资料，并在切换时写入当前用户的 `.codex` 目录后启动默认 Codex。因此参考时应选择基础可靠性、账号状态和错误归因能力，不应直接搬入重型反代、远程部署或多实例体系。

## 可参考的工程做法

### 1. 配置文件原子写入和备份恢复

参考项目在账号存储层做了临时文件写入、替换、私有权限、滚动备份和损坏恢复。

CodexMultiLauncher 原先直接覆盖写入 `config.json`，如果写入过程中断电、杀进程或磁盘异常，可能造成配置损坏。

适合借鉴的点：

- 保存配置时先写入临时文件。
- 临时文件写入成功后再替换正式配置。
- 保留最近两份可恢复备份。
- 主配置损坏时优先从备份恢复。
- 恢复前保留损坏文件，方便排查。

### 2. 配置损坏后的自动恢复

参考项目在读取存储失败时不会立即重置，而是尝试从备份文件和残留文件恢复。

CodexMultiLauncher 可以采用更轻量的策略：

- 主配置读取失败时备份为 `config.json.corrupt`。
- 依次尝试 `config.json.last-good.json` 和 `config.json.prev-good.json`。
- 备份可用时自动写回主配置。
- 所有恢复失败时才使用默认配置。

### 3. 账号目录健康检查

参考项目会记录账号授权文件、配置文件和 profile 状态，用于提示账号是否完整。

CodexMultiLauncher 不需要完整账号模型，但可以检查每个账号的基础运行条件：

- 账号目录是否存在。
- `auth.json` 是否存在。
- `config.toml` 是否存在。
- 账号目录是否可写。
- 当前活动 `.codex` 目录是否具备可写条件。

### 4. 启动失败时的排障信息

参考项目有环境诊断和状态聚合思路。

CodexMultiLauncher 不再保留独立诊断页面，但可以在启动失败、账号异常或调试命令中复用轻量诊断信息，帮助用户定位常见问题：

- 配置文件是否存在。
- 配置备份是否存在。
- 账号根目录是否可写。
- 当前活动 `auth.json` 和 `config.toml` 是否存在。
- Codex 桌面程序、微软商店应用标识或 `codex` 命令是否可启动。
- 每个账号的账号资料是否完整。

这类诊断适合做成错误弹窗中的附加信息或开发调试命令，不需要恢复为单独页面。

### 5. 账号状态展示

CodexMultiLauncher 可以继续借鉴参考项目的状态展示方式，但状态只围绕当前轻量账号模型：

更合适的账号行状态包括：

- “运行中”：当前激活账号且检测到 Codex 进程。
- “就绪”：账号未运行但可操作。
- “认证已保存”：账号资料中存在 `auth.json`。
- “认证缺失”：账号资料不完整，需要重新保存或修复。

这样用户不用等启动失败后才知道账号资料是否完整。

### 6. 更细的错误归因

参考项目对很多错误做了分类，例如授权失效、文件损坏、端口占用、依赖缺失。

CodexMultiLauncher 可以在不增加复杂度的前提下，把常见问题转成更明确的中文提示：

- 微软商店版路径失效。
- 账号目录被占用。
- 账号目录不可写。
- 账号认证文件缺失。
- 当前活动 `.codex` 目录不可写。
- 默认配置文件不存在。

### 7. 路径识别失败时展示诊断详情

CodexMultiLauncher 已经支持通过应用包信息、WindowsApps 扫描和运行进程识别 Codex 路径。

可以继续参考参考项目的诊断思路，在自动识别失败时给出更多上下文，而不是只提示“未找到 Codex 程序”：

- 当前记录的路径。
- 该路径是否存在。
- WindowsApps 是否可访问。
- 微软商店应用标识是否能读取。
- `codex` 命令是否可执行。
- 当前是否存在运行中的 Codex 进程。

## 暂不建议引入的能力

以下能力虽然参考项目实现完整，但不适合当前阶段引入 CodexMultiLauncher：

### 本地 API 反代

这会把项目从“多账号桌面启动器”扩展成“账号池代理服务”，需要鉴权、端口监听、协议转换、模型过滤、请求日志和安全边界设计。

当前不建议引入。

### 会话成本统计

当前项目已经有账号额度刷新能力，但不建议继续扩展到会话成本统计。成本统计需要扫描 Codex 会话日志、解析 token 事件、维护价格或模型映射，并处理日志格式变化，会明显扩大维护范围。

### 远程 Linux 部署

远程部署涉及 SSH、密钥、密码、服务安装、远程日志和权限处理，安全边界复杂。

当前项目只面向 Windows 本机启动，不建议引入。

### Cloudflare 隧道

公网暴露会引入额外风险，且和本项目的“本地隔离启动”目标不一致。

当前不建议引入。

### 用 Tauri 重写

除非后续需求显著扩大，否则不建议重写技术栈。

## 后续建议

后续如果继续参考 `codex-tools-2.0.5`，建议仍按“轻量、稳定、可排障”的方向推进：

1. 启动失败时附带账号健康检查结果。
2. 给配置恢复流程增加一次性提示，让用户知道已从备份恢复。
3. 将常见异常文案进一步归类，减少裸异常文本。
4. 增加账号导出、删除前备份和从 `auth.json` 导入账号。
5. 基于额度缓存推荐更适合启动的账号。
